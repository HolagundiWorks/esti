A mid-size studio with twelve active projects might run twelve parallel feedback loops:
WhatsApp for one client, email for another, marked-up PDFs for a third, site calls for
a fourth. The architect becomes a **router** Ś copying, reconciling, and apologising
for version confusion.

Efficiency is not about saying no faster. It is about **one authoritative loop per
project** that clients and consultants can learn once.

## Why multiple loops multiply error

Each channel has different failure modes:

- **WhatsApp** Ś fast, but revisions are not tied to drawing numbers; context disappears in scrollback.
- **Email** Ś attachments duplicate; "Re: Re: Fwd:" threads hide the latest approved set.
- **Verbal site meetings** Ś decisions without minutes evaporate.
- **Portal-less PDF mark-up** Ś redlines on files that are not the issued revision.

Staff spend hours reconciling "what the client actually meant" across channels before
they draw.

## The efficient model: one front door per project

1. **Issued drawings** go through **transmittals** Ś numbered, dated, attributed.
2. **Change requests** log on the **project decision register** with category and fee impact.
3. **Clients** use a **scoped portal** for approvals and status Ś not mixed personal chat.
4. **Consultants** see only their engagement Ś RFIs and issued sets, not the whole office.
5. **Internal coordination** stays in **tasks** linked to phases Ś not side threads.

External channels still exist for relationship warmth. The **record** lives in one place.

## Routing rules that offices adopt

| Incoming feedback | Route to |
|-------------------|----------|
| Client design change | Decision register ? approval ? drawing revision |
| Consultant RFI | Project consultant engagement ? task or decision |
| Site instruction | Site module ? linked drawing sheet if applicable |
| Fee / payment query | Finance thread on client record Ś not project chat |

Publish these rules in your client onboarding pack. Reference the
[AORMS wiki](https://aorms.in/wiki/how-to-use-aorms) for client portal setup.

## Technology is not the whole answer

Software enables the model; **office discipline** makes it stick:

- Principals model the behaviour Ś no "just WhatsApp me the PDF."
- Project architects close loops Ś every open client comment maps to a decision state.
- Weekly sweep Ś zero high-impact decisions in "draft" near issue dates.

AORMS connects portals, decisions, drawings, and tasks on one cloud workspace so the
sweep is a filtered view, not a meeting.

## Frequently asked questions

### Clients insist on WhatsApp Ś what then?

Acknowledge on WhatsApp; **log in the system** within the same day. ESTI can help
draft formal revision requests from meeting minutes.

### Multiple stakeholders on the client side?

Portal accounts per contact, scoped to the same project. Transmittals show who received
which revision.

### Consultants on WeTransfer?

Same rule Ś receipt is logged; files attach to the drawing register row.
